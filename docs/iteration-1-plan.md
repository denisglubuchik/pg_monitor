# План итерации 1

## Название

FastAPI каркас + конфиг + structured logging.

## Цель итерации

Подготовить базовый runtime сервиса `pg-monitor` для дальнейших итераций:
- поднимается FastAPI-приложение;
- доступен endpoint `/healthz`;
- работает загрузка конфигурации с правильным приоритетом источников;
- включено структурированное JSON-логирование с маскированием секретов.

## Scope итерации

1. Каркас приложения и базовая структура пакета.
2. `settings loader` с приоритетом:
`defaults < .env < OS env`.
3. Валидация обязательных параметров (включая DSN).
4. Structured JSON logging:
- настраиваемый `log_level`;
- маскирование секретных значений.
5. Endpoint `/healthz` и минимальный smoke runtime.

## Out of Scope

1. SQL collector и опрос `pg_stat_*`.
2. Сохранение snapshot и delta-аналитика.
3. Endpoint `/metrics` и Prometheus-интеграция.
4. Grafana, Alertmanager, Loki/Promtail.

## Definition of Ready (перед кодом)

1. Согласован формат конфига (минимум обязательных полей и имена env).
2. Согласован список полей, которые считаются секретами и подлежат masking.
3. Согласованы критерии готовности этой итерации.
4. Нет открытых неуточненных вопросов по архитектуре итерации 1.

## Детальный план работ

1. Подготовка структуры проекта.
Создать `src/pg_monitor/` с модулями `config`, `logging`, `api` и точкой входа приложения.

2. Тесты на конфигурацию (до реализации).
Добавить тесты для:
- приоритета источников конфигурации;
- падения валидации при отсутствии обязательного DSN;
- корректной загрузки валидного конфига.

3. Тесты на логирование (опционально).
Можно отложить на следующую итерацию, если логирование реализовано и проверено вручную.

4. Тесты на API (до реализации).
Добавить тест `GET /healthz -> 200` и базовую проверку тела ответа.

5. Реализация конфигурации.
Сделать settings loader с поддержкой:
- defaults;
- `.env`;
- OS env;
с приоритетом согласно требованиям.

6. Реализация логирования.
Сделать инициализацию structured logging с единым форматтером JSON и фильтром masking.

7. Реализация FastAPI каркаса.
Поднять приложение, подключить health router и стартовую инициализацию логирования/конфига.

8. Локальная проверка.
Прогнать тесты, поднять сервис локально и проверить `/healthz`.

## Критерии готовности (Definition of Done)

1. Написаны и проходят unit-тесты на:
- приоритет конфигов;
- `/healthz`.
2. Сервис стартует локально без ручных правок кода.
3. `/healthz` стабильно возвращает успешный ответ.
4. Логи пишутся в JSON и не содержат не замаскированных секретов.

## Артефакты итерации

1. Код каркаса сервиса и модулей конфигурации/логирования.
2. Набор автотестов для итерации 1.
3. Краткая заметка о фактически реализованной схеме конфига (поля, источники, приоритет).

## Риски и меры

1. Риск: неоднозначность формата конфиг-файла.
Мера: зафиксировать `.env` как единый конфиг-источник и покрыть тестом загрузки.

2. Риск: неполное masking секретов.
Мера: явный список паттернов и тесты на redaction.

3. Риск: дрейф структуры проекта от архитектуры.
Мера: сразу создать директории в соответствии с `docs/architecture.md`.
